Input Relay Project Brief

A local-first relay that lets a verified phone edit a PC-side proxy buffer and send the final text into the currently focused input on the PC.

Product Shape

The app is not a remote keyboard. The phone edits an app-owned proxy buffer. Only explicit allowlisted actions can affect the PC, and insertion into other apps happens as one deliberate action.

What it does

  • Phone connects directly to the PC relay through Tailscale.
  • Phone edits a shared proxy buffer in real time.
  • A verified phone can trigger Accept if granted permission.
  • Accept tries to insert the final buffer into the currently focused PC input.
  • If insertion fails, the proxy window comes to the front for manual copy and the phone gets a toast.

What it avoids

  • No cloud relay and no public Tailscale Funnel.
  • No native phone app for the initial product.
  • No arbitrary phone-driven keyboard control on the PC.
  • No PC mouse-click Accept by default, because it steals focus.
  • No text contents in logs.

Architecture

The background service is the authority. Tauri is a local admin and proxy UI. The phone PWA is served by the PC relay, so the client is always close to the machine that will perform the action.

Phone PWA React, passkey auth, WebSocket buffer editing, action buttons.

Adapter boundaries

  • Admin IPC adapters
    How the local desktop UI controls the service.
  • Input backend adapters
    How the service performs narrow actions such as AcceptInsertText or LockPc.
  • Keychain adapters
    KWallet or Secret Service on this PC; platform credential stores later.
  • Mock backend
    For debug and tests only; normal use needs a real supported backend.

Core Flows

These flows are the product contract. Implementation can change internally, but these user-facing behaviors should stay stable.

First run

  1. Detect KDE Wayland and required tools.
  2. Create app password for fallback recovery.
  3. Set up OS keychain unlock by default.
  4. Initialize full encrypted SQLite database.
  5. Check Tailscale and show Serve command.
  6. Register first phone from the PC.

Phone registration

  1. PC opens Register Device.
  2. PC creates short-lived QR pairing session.
  3. Phone scans QR and opens the PWA.
  4. Phone registers a passkey for the PC origin.
  5. PC approves the device and grants permissions.

Accept

  1. Phone edits the proxy buffer live.
  2. Verified phone taps Accept.
  3. Service checks device permission.
  4. Backend inserts into the currently focused PC input.
  5. Success clears the buffer; failure keeps it and shows manual copy.

Security Model

Access is layered: private network transport, passkey device auth, app unlock, encrypted local state, per-device permissions, and narrow action handlers.

Decisions

  • Tailscale is required for the default path; both devices must be in the tailnet.
  • Tailscale Serve is allowed; Funnel is not part of the plan.
  • Passkeys are default for verified phones.
  • Registration is initiated from the PC only.
  • Permissions are assigned per device.
  • App state is locked until keychain or app password unlocks it.

Local protection

  • Whole SQLite database encryption via SQLCipher and rusqlite if packaging is workable.
  • OS keychain is the default unlock helper on supported systems.
  • App password remains the fallback and recovery path.
  • Linux admin IPC uses Unix socket plus peer credential checks.
  • Locked phone page may load only a minimal "relay locked" state.
  • Logs must never include buffer text, accepted text, clipboard text, or history entries.

Action Protocol

Phone messages should be structured and versioned. The MVP can use simple snapshots, but the protocol should leave room for richer edits later. 

{
  "version": 1,
  "type": "buffer.setText",
  "text": "Final text from phone",
  "selection": { "start": 21, "end": 21 }
}
{
  "version": 1,
  "type": "action.acceptInsertText",
  "requestId": "01J...",
  "bufferRevision": 42
}
Important: buffer editing actions operate only on the app-owned proxy buffer. PC actions such as AcceptInsertText and LockPc are explicit commands with backend capability checks.

Runtime Rules

These rules keep the behavior predictable under normal use, failure, multiple devices, and privacy-sensitive states.

Session One active phone/device session at a time. A second verified phone gets read-only status and cannot edit or Accept.
History Modes are none, last X, and all. Default is none. If last X is selected, X defaults to 10.
Buffer after Accept Successful insertion clears the buffer. Failed insertion keeps it unchanged for manual copy.
Locked state No buffer, history, devices, permissions, registration, or actions are available until app unlock.
Unsupported system Inform the user clearly that normal operation is unsupported. Mock backend remains available for debug only.
Installer First installer targets this KDE Wayland PC and is structured so other environments can be added later.

Implementation Risks

These are not product decisions. They are engineering investigations that should be handled early because they can affect the installer and backend boundaries.

KDE Wayland insertion

Find the safest reliable route for AcceptInsertText. Clipboard plus paste trigger is allowed internally if it preserves clipboard when possible and never exposes arbitrary keyboard control.

SQLCipher packaging

Confirm build/install story for SQLCipher with Rust on this machine before depending on it deeply.

Passkeys and Tailscale origin

Passkeys need a secure stable origin. Tailscale MagicDNS plus HTTPS is the expected default, and the app should check instead of silently configuring Serve.

KWallet unlock behavior

This PC has KWallet/Secret Service tooling available. The keychain adapter still needs careful behavior when the wallet is locked, unavailable, or cleared.
